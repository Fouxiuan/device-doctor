## 一、Demo 简介

### 是什么

一个基于 Web 的轻量级设备异常处理工作台，把「设备异常状态看板」和「结构化智能知识库」深度绑定，让设备异常处理从「人追着问题跑」变成「数据推着人走」。

- **形态：** 纯前端单页应用（HTML5 + CSS3 + 原生 JavaScript，无任何框架依赖）
- **存储：** LocalStorage 本地持久化 + JSON 一键导入导出
- **AI：** 接入豆包大模型（火山引擎方舟平台），提供异常智能诊断与方案生成

### 面向谁

硬件测试工程师、设备运维人员、质检流程专员，以及任何需要「跟进故障处理进度 + 沉淀解决方案」的岗位。

### 主要功能

1. **可视化 Kanban 看板** — 5 列覆盖处理全流程（数据校对 → 检测中 → 判定正常/危险 → 穿戴测试），卡片支持拖拽流转和键盘方向键移动，支持任意方向回退
2. **结构化智能知识库** — 分类目录 + 标签筛选 + 关键词秒级检索，Markdown 实时双栏预览编辑，方案状态标识（已验证/待完善/高风险）
3. **AI 智能诊疗** — 一键调用大模型，基于异常信息自动推荐标签、阶段、可能原因和处理建议，降低人工判断成本
4. **完整数据闭环** — 发现异常 → 指派责任人 → 追踪状态 → 沉淀方案 → 再次检索，全链路打通

## 二、Demo 创作思路

### 灵感来源

这完全是我测试岗的真实日常。我受够了每天在 Excel 里手动更新状态、在文件夹里大海捞针找旧案例。设备数据异常时，处理状态（校对/检测/穿戴）全靠口头沟通或手动改 Excel，信息严重滞后；发现问题后找对应责任人要翻通讯录，沟通成本高；历史异常解决经验散落在聊天记录或本地文档里，下次遇到同类问题依然要重新推理，无法即查即用。

既然工作避免不了处理异常，不如自己造个工具。

### 想解决的问题

| 痛点 | 现状 | 解决方案 |
|------|------|----------|
| 状态靠吼 | 设备处理状态全靠口头沟通或手动改 Excel，信息严重滞后 | 可视化 Kanban 看板，一屏总览所有设备阶段 |
| 找人靠翻 | 不知道异常对应谁负责，翻通讯录沟通成本高 | 异常记录直接绑定责任人，看板卡片一目了然 |
| 经验靠脑 | 历史解决经验散落在聊天记录或本地文档，无法即查即用 | 结构化知识库，标签化归档，关键词秒级检索 |

### 为什么做这个方向

- **真实价值优先：** 大赛不拼复杂度，只看真实价值。这是我每天会用到的工具，痛点真实、场景高频。
- **小切口深挖掘：** 不做大而全的设备管理平台，只聚焦「异常处理闭环」这一个高频场景，把流程做透。
- **AI 锦上添花而非喧宾夺主：** 核心是流程工具，AI 在关键节点（异常诊断、方案生成）提供增量价值，不依赖 AI 也能完整使用。

## 五、技术亮点

- **纯前端零依赖：** 无框架、无后端、无构建工具，HTML+CSS+原生 JS，打开即用
- **双存储策略：** LocalStorage 日常持久化 + JSON 导入导出做数据迁移备份
- **原生 Drag and Drop API：** 实现流畅的 Kanban 拖拽，支持任意方向流转含回退
- **AI 能力落地：** 豆包大模型驱动异常智能诊断，推荐标签/阶段/原因/建议一键应用
- **组件化架构：** IIFE 模块模式，9 个 JS 模块职责清晰，代码可维护
- **无障碍优先：** 焦点陷阱、键盘可操作、ARIA 语义、响应式 7 档断点

## 六、开发心得

这次开发最深的体会是「人机协作」的边界。TRAE 负责把重复性工作（生成骨架、写样板代码、批量替换、可访问性审查）快速完成，我负责把控产品方向、交互细节和真实场景适配。比如：

- **webview 兼容问题：** 原生 `confirm()` 在 TRAE IDE webview 中触发 React error #185，TRAE 帮我快速定位并实现了自定义 Dialog 替代方案
- **SVG 裁切细节：** 爱心图标因 viewBox 与描边路径不匹配被裁切，通过 `overflow: visible` 和改用标准 Lucide 图标解决
- **交互习惯对齐：** 知识库分类栏从小箭头点击改为整栏可点击，Markdown 回车改为 Word 式单换行，都是基于真实使用反馈的迭代

这些细节单靠 AI 想不到，但有了 AI 的执行效率，我能把更多精力放在体验打磨上。

## 七、我和 TRAE 配合的真实感受

说实话，报名这个比赛之前，我心里是打鼓的。我做测试岗，日常和设备、数据、异常打交道，脑子里装满了"要是有个工具能这样就好了"的想法，但真要自己动手写一个，以前觉得那是程序员的事，离我很远。

打开 TRAE IDE 的那一刻，我做的第一件事不是写代码，而是**像个跟同事描述需求一样，把每天的痛点倒给了它**。我说："设备状态全靠吼，找人全靠翻，经验全靠脑，我想做个看板把这些串起来。"我以为它会让我先学一堆术语，结果它直接问我："那你看板想分几列？"——就这一句，我突然觉得，哦，原来做产品可以从这里开始。

最让我有「原来这么简单」那个瞬间的，是看板拖拽。我跟它说"卡片要能拖到别的列，状态自动变，还得能往回拖"，它就把原生 Drag and Drop API 的逻辑写好了。我盯着那段代码看了很久——这个功能我以前觉得是"高级功能"，结果在 TRAE 里就是一句话的事。那天晚上我把整个看板跑通，拖来拖去玩了好久，像个小孩拿到了新玩具。

但 TRAE 不是万能的，它也有"搞不定"的时候，而那些时刻恰恰是我最有参与感的时刻。比如爱心图标被裁切了一半，它第一次给的方案没解决，我又描述了一遍"是描边超出 viewBox 被切了"，它才反应过来加 `overflow: visible`。再比如原生 `confirm()` 在 webview 里直接报 React 错误，我俩都没料到，是一起排查、一起试错，最后它帮我写了套自定义 Dialog 才绕过去。**这种"一起踩坑再一起爬出来"的过程，让我觉得自己不是在用工具，而是在合作。**

它帮我跨过的最大的坎，是 AI 能力接入。我一直想给异常记录加个"智能诊断"，但调用大模型 API 这事，以前对我来说就是个黑盒。我跟 TRAE 说"我想接豆包，根据异常信息自动给标签和建议"，它就帮我把 `ai.js` 模块从零搭起来，API 调用、prompt 设计、结果渲染、一键应用，一步步带我走完。第一次看到大模型真的返回了"可能原因"和"处理建议"的那一刻，我有点恍惚——**这真的是我一个人做出来的吗？**

回头看这一个多月的开发，我最真实的感受是：**TRAE 没有替我成为程序员，但它让我成为了一个能把想法落地的人。** 那些痛点不再只是下班路上脑子里的一句抱怨，而是变成了打开浏览器就能用的工具。同事们问我"这玩意儿哪下载的"，我说"我自己做的"——这个回答带来的底气，是 TRAE 给我的。

如果说以前觉得"造个工具"是别人的故事，那现在我觉得，**只要你能把问题讲清楚，TRAE 就能陪你把它做出来**。这大概就是 AI 时代给普通人最好的礼物吧。
