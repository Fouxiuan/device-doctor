## 1. 项目概述

### 1.1 创意名称
**设备诊疗师 —— 异常闭环与知识管家**

### 1.2 一句话描述
一个基于 Web 的轻量级设备异常处理工作台，将「状态看板」与「智能知识库」深度绑定，让异常处理从「人追着问题跑」变成「数据推着人走」。

### 1.3 解决的核心痛点

| 痛点 | 现状 | 解决方案 |
|------|------|----------|
| 状态靠吼 | 设备处理状态全靠口头沟通或手动改 Excel，信息严重滞后 | 可视化 Kanban 看板，一屏总览所有设备阶段 |
| 找人靠翻 | 不知道异常对应谁负责，翻通讯录沟通成本高 | 异常记录直接绑定责任人，看板卡片一目了然 |
| 经验靠脑 | 历史解决经验散落在聊天记录或本地文档，无法即查即用 | 结构化知识库，标签化归档，关键词秒级检索 |

### 1.4 目标用户
- 硬件测试工程师
- 设备运维人员
- 质检流程专员
- 任何需要「跟进故障处理进度 + 沉淀解决方案」的岗位

### 1.5 产品定位
**个人优先，团队可扩展。** Demo 阶段为个人效率工具（纯前端 + LocalStorage），通过 JSON 导入导出实现数据分享。团队多人协作（用户系统、实时同步）作为后续扩展方向，在参赛展示中作为产品愿景呈现。

## 2. 核心功能设计

### 2.1 功能架构

```
┌─────────────────────────────────────────┐
│           设备诊疗师 (单页应用)            │
├─────────────┬─────────────┬─────────────┤
│   状态看板   │   知识库    │   弹窗组件   │
│  (Dashboard) │ (Knowledge) │   (Modal)   │
├─────────────┴─────────────┴─────────────┤
│         数据层 (LocalStorage + JSON)      │
└─────────────────────────────────────────┘
```

### 2.2 状态看板（首页）

**功能描述：**
- 顶部统计卡片：展示 5 个阶段的设备数量（数据校对 / 检测中 / 判定正常 / 判定危险 / 穿戴测试）
- Kanban 看板：横向 5 列，每列对应该阶段，卡片可拖拽切换状态
- 设备卡片：显示设备编号、异常简述、负责人、时间
- 操作按钮：新增异常（弹窗）、导出数据、导入数据

**交互细节：**
- 点击统计卡片可筛选看板，只显示该阶段设备
- 卡片拖拽到另一列时，自动更新设备状态（**支持任意方向移动，包括回退**，如「判定危险」维修后回到「检测中」）
- 点击卡片可查看详情或编辑

### 2.3 智能知识库

**功能描述：**
- 搜索框：支持关键词模糊匹配，实时联想历史异常标签
- 标签筛选：常用异常类型快速筛选（温度传感器、电压异常、固件问题等）
- 知识卡片：展示方案标题、关联设备、解决次数、处理步骤摘要
- 状态标签：已验证 / 待完善 / 高风险，帮助判断方案可靠性
- 互动数据：点赞数、讨论数，反映方案实用程度

**交互细节：**
- 搜索时同时匹配标题、标签、设备编号
- 点击标签自动添加到筛选条件
- 卡片展开显示完整处理步骤

### 2.4 新增异常弹窗

**功能描述：**
- 设备编号（支持输入联想）
- 异常现象简述
- 当前阶段（下拉选择）
- 负责人（支持 @ 提及）
- 异常标签（多选，支持自定义）
- 详细描述 / 现场记录
- 保存并创建 / 保存并关联知识库

**交互细节：**
- 「保存并关联知识库」：若检测到相似历史异常，自动推荐关联方案
- 标签选择后自动同步到知识库标签体系
- **AI 优化记录**：用户手动填写解决过程后，可通过 TRAE IDE 的 MCP 能力调用大模型，对记录内容进行结构化优化（补充步骤、修正表述、提取关键词），优化后的内容存入知识库

### 2.5 数据闭环流程

```
发现异常 → 指派责任人 → 追踪状态 → 沉淀方案 → 再次检索
  (弹窗)      (@提及)     (看板)      (知识库)     (搜索)
```

## 3. 技术架构

### 3.1 技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| 结构 | HTML5 | 语义化标签 |
| 样式 | CSS3 | Flexbox/Grid 布局，CSS 变量管理主题色 |
| 逻辑 | 原生 JavaScript (ES6+) | 无框架依赖，降低复杂度 |
| 存储 | LocalStorage | 主存储，日常操作 |
| 备份 | JSON 文件导入/导出 | 数据持久化与分享 |
| 交互 | 原生 DOM API + Drag and Drop API | 看板拖拽 |

### 3.2 文件结构

```
/device-doctor/
├── index.html              # 主入口，路由容器
├── css/
│   ├── main.css            # 全局样式、CSS 变量、布局框架
│   ├── dashboard.css       # 看板样式（统计卡片、Kanban 列、设备卡片）
│   ├── knowledge.css       # 知识库样式（搜索框、标签、知识卡片）
│   └── modal.css           # 弹窗样式（表单、动画）
├── js/
│   ├── app.js              # 路由管理、页面初始化、全局事件
│   ├── store.js            # LocalStorage 封装、JSON 导入导出
│   ├── dashboard.js        # 看板渲染、拖拽逻辑、筛选交互
│   ├── knowledge.js        # 知识库搜索、标签过滤、卡片渲染
│   ├── modal.js            # 弹窗组件（新增/编辑异常）
│   └── data.js             # 示例数据（Demo 演示用）
└── assets/
    └── (图标、图片等资源)
```

### 3.3 数据模型

#### 设备异常记录 (Issue)
```javascript
{
  id: "ISS-20260622-001",
  deviceId: "A-1023",
  title: "温度传感器漂移",
  description: "设备运行 2 小时后温度读数偏差超过 5°C",
  stage: "detecting",        // verifying | detecting | normal | danger | wearing
  assignee: "张三",
  tags: ["温度传感器", "校准"],
  createdAt: "2026-06-22T09:00:00",
  updatedAt: "2026-06-22T14:30:00",
  knowledgeId: "KNW-001"     // 关联知识库方案（可选）
}
```

#### 知识库方案 (Knowledge)
```javascript
{
  id: "KNW-001",
  title: "温度传感器漂移 — 校准与更换流程",
  tags: ["温度传感器", "校准"],
  relatedDevices: ["A-1023", "A-1056"],
  solveCount: 8,
  status: "verified",        // verified | pending | high-risk
  content: "1. 进入工程模式 → 2. 执行零点校准 → 3. 对比标准源误差...",
  author: "张三",
  createdAt: "2026-06-15",
  likes: 12,
  comments: 3
}
```

### 3.4 存储策略

**数据模型约束：**
- 一个设备同一时间只有一条活跃的异常记录
- 异常解决后，设备可产生新的独立异常记录
- 每条 Issue 独立流转，互不影响

**LocalStorage 结构：**
```
key: "device-doctor-data"
value: {
  issues: [...],      // 异常记录数组
  knowledge: [...],   // 知识库数组
  tags: [...],        // 标签体系（去重）
  settings: {...}     // 用户设置
}
```

**JSON 导入/导出：**
- 导出：将 LocalStorage 数据序列化为 JSON 文件下载
- 导入：读取 JSON 文件，校验格式后合并或替换本地数据
- 格式与 LocalStorage 结构一致，便于迁移

## 4. 页面设计

### 4.1 状态看板页

**布局：**
- 顶部导航栏：Logo + 页面切换（看板 / 知识库）+ 操作按钮
- 统计卡片行：5 个等宽卡片，颜色区分阶段
- Kanban 区域：5 列横向滚动，每列顶部显示阶段名称和数量
- 设备卡片：白色卡片，左侧彩色边框标识阶段，内容紧凑

**响应式：**
- 桌面端：5 列完整展示
- 平板端：横向滚动，每列最小宽度 280px
- 移动端：单列堆叠，卡片纵向排列

### 4.2 知识库页

**布局：**
- 搜索区域：大搜索框 + 搜索按钮 + 新增方案按钮
- 标签筛选区：横向滚动标签列表，点击切换筛选状态
- 知识列表区：纵向卡片列表，每张卡片包含标题、元信息、摘要

**空状态：**
- 无搜索结果时显示提示：「未找到相关方案，试试其他关键词？」
- 知识库为空时显示引导：「暂无方案，处理完异常后记得沉淀经验！」

### 4.3 弹窗组件

**布局：**
- 遮罩层：半透明黑色背景，点击可关闭
- 弹窗容器：居中，最大宽度 600px，圆角白色背景
- 表单内容：纵向排列，标签左对齐，输入框占满宽度
- 底部操作区：取消 / 保存并创建 / 保存并关联知识库

**动画：**
- 打开：遮罩淡入 200ms，弹窗从上方滑入 300ms
- 关闭：反向动画

## 5. 交互设计

### 5.1 看板交互

| 操作 | 反馈 |
|------|------|
| 点击统计卡片 | 看板筛选，只显示该阶段卡片；再次点击取消筛选 |
| 拖拽卡片到另一列 | 卡片移动，目标列高亮提示，松手后状态更新 |
| 点击卡片 | 展开详情或进入编辑弹窗 |
| 点击「新增异常」 | 打开弹窗，表单为空 |

### 5.2 知识库交互

| 操作 | 反馈 |
|------|------|
| 输入搜索关键词 | 实时联想标签，回车或点击搜索执行匹配 |
| 点击标签 | 标签高亮，列表过滤显示包含该标签的方案 |
| 点击「展开全文」 | 卡片展开显示完整处理步骤 |
| 点击「新增方案」 | 打开弹窗，从当前异常记录自动填充部分字段 |

### 5.3 弹窗交互

| 操作 | 反馈 |
|------|------|
| 输入设备编号 | 联想历史设备，下拉提示 |
| 选择异常标签 | 标签变为选中状态，支持多选 |
| 点击「保存并关联知识库」 | 检测相似标签/标题，推荐关联方案；若无匹配则提示创建新方案 |

## 6. 数据流设计

### 6.1 新增异常流程

```
用户点击「新增异常」
    ↓
打开 Modal，用户填写表单
    ↓
点击保存
    ↓
store.js 写入 LocalStorage
    ↓
dashboard.js 重新渲染看板
    ↓
若选择「关联知识库」→ knowledge.js 更新知识库列表
```

### 6.2 状态变更流程

```
用户拖拽卡片从「检测中」到「判定正常」
    ↓
dashboard.js 捕获拖拽事件
    ↓
更新 issue.stage 字段
    ↓
store.js 写入 LocalStorage
    ↓
重新渲染两列（源列和目标列）
    ↓
统计卡片数字更新
```

### 6.3 数据导出流程

```
用户点击「导出数据」
    ↓
store.js 读取 LocalStorage 全部数据
    ↓
序列化为 JSON 字符串
    ↓
创建 Blob，触发浏览器下载
    ↓
下载文件: device-doctor-backup-20260622.json
```

## 7. 非功能需求

### 7.1 性能
- 首屏加载时间 < 2s（纯前端，无网络请求）
- 看板渲染 50 条记录无卡顿
- 搜索响应 < 100ms

### 7.2 兼容性
- 支持 Chrome / Edge / Firefox / Safari 最新两个版本
- 移动端浏览器可用（响应式布局）

### 7.3 可访问性
- 表单元素关联 label
- 按钮和链接有明确的焦点样式
- 颜色不作为唯一信息传递手段（图标 + 文字）

## 9. 参赛展示要点

### 9.1 Demo 演示路径
1. **开场**：打开看板，展示一屏总览的便利性
2. **痛点还原**：「昨天这个设备还在检测中，今天突然找不到了」→ 展示筛选和搜索
3. **核心闭环**：新增异常 → 指派责任人 → 拖拽变更状态 → 沉淀知识库 → 下次检索复用
4. **数据安全**：演示导出功能，「数据永远属于你自己」

### 9.2 技术亮点
- 纯前端实现，零后端依赖，打开即用
- LocalStorage + JSON 双存储策略，兼顾体验与数据安全
- 原生拖拽 API 实现 Kanban 交互
- 组件化架构，代码清晰易维护
